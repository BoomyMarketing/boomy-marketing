# SEO + GEO Strategy 2026-2027

**Last updated:** 2026-05-23
**Status:** Living document ‚Äî update as Google + LLM behavior shifts
**Scope:** Canonical, niche-agnostic framework. Drop into any project. Replace `{niche}`, `{primary_keyword}`, `{city}`, `{expert_name}`, `{credentials}` with the per-site values.

> **North star** ‚Äî every page must win attention in **two parallel surfaces** that increasingly overlap:
> 1. **Classic Google SERPs** ‚Äî organic + local pack (where relevant).
> 2. **AI surfaces** ‚Äî Google AI Overviews, ChatGPT browse, Perplexity, Bing Copilot, Claude search.
>
> We optimize the same page for both. Every page must be **citation-ready**: short paragraphs, declarative answers, schema-rich, fast-loading.

## 1. What changed in 2025‚Äì2026

These are the eight shifts that drive every other decision in this document. If a page does not respond to all eight, it is a 2024-era page.

| # | Shift | What it means | Our response |
|---|-------|---------------|--------------|
| 1 | **Calibrated word-count targets, completeness > length** | Google ranks topic completeness, not raw word count. | 2 | **Multi-tier KW strategy + read-aloud test, persona modifiers** | Keyword stuffing is dead. KW density tools mislead. | 3 | **AI citations driven by third-party mentions across surfaces** | LLMs cite brands that other reputable sites mention, not just brands that publish a lot. | Active PR + citation building: industry directories, niche communities, Reddit, local news, partner sites. | 4 | **Share of Voice in AI (Overviews + ChatGPT + Perplexity)** | New visibility KPI replaces "ranking #1". | 6 | **Standalone paragraphs (LLM extracts individually)** | LLMs extract paragraphs, not full pages. ## 3. The 4-tier keyword model

For every indexable page we define exactly four tiers of keywords:

```
Primary KW       ‚Äî 1     ‚Äî title, H1, URL, first 100 words, hero img alt
Secondary KWs    ‚Äî 3‚Äì5   ‚Äî H2/H3 headings, intro of each section
Supporting KWs   ‚Äî 8‚Äì15  ‚Äî body, FAQ, schema, image alts
Persona modifiers ‚Äî 2‚Äì4  ‚Äî phrasing variations matching audience
```

### Density guidance

- **Primary KW:** 1.5‚Äì2.5% (e.g. ~30 occurrences in 1,500 words). Never higher.
- **Secondary KWs:** 0.8‚Äì1.2% each.
- **Supporting KWs:** appear once or twice, contextually only.
- **Persona modifiers:** sprinkle through body where they sound natural.

**Override rule:** the read-aloud test wins. If a passage sounds stuffed when read aloud, it is stuffed ‚Äî regardless of what density tools say.

### Cannibalization rule

**One SERP cluster = one page maximum.** If two keyword clusters have ‚â•60% URL overlap on Google SERP, merge them into the same page. Use DataForSEO to verify SERP similarity before assigning.

| Primary | `{primary_keyword}` | `<title>`, `<h1>`, URL slug, first sentence, og:title, hero img alt |
| Secondary | `emergency {primary_keyword}` | H2 "Emergency Service" + intro |

## 4. The 5 persona archetypes (universal)

These archetypes work for any niche. Adapt the trigger queries and copy notes to your industry ‚Äî the underlying behavioral pattern stays the same.

| # | Persona | Universal pattern | How to adapt to your niche |
|---|---------|------------------|----------------------------|
| 1 | **Stressed user ‚Äî emergency** | Acute problem, urgent. Time-sensitive. Searches in panic, often after-hours. | Replace with: homeowner with broken HVAC, parent with sick child needing clinic, business owner with crashed server. Copy: 24/7 availability, dispatch time, reassurance, "we answer when you call". |
| 2 | **Stressed user ‚Äî broken thing** | Specific item failure. Less urgent than #1 but still pain-driven. Wants price range + safety. | Replace with: broken appliance, leaking pipe, error code on device, failing component. Copy: price range, safety warning, why DIY is dangerous/risky, when to call a pro. |
| 3 | **Planner ‚Äî researching replacement / new purchase** | Daytime browsing. Comparing options. Wants depth, brand comparison, ROI, financing. | Replace with: home buyer, software evaluator, equipment shopper. Copy: buying guide depth, brand vs brand comparison, ROI / TCO / financing, social proof. |
| 4 | **Cost-checker** | Pre-decision. Wants pricing tables, ranges with explanations, no-surprise pledge. | Replace with: any user researching cost before commitment. Copy: structured price ranges, "what affects the price" section, no-surprise / transparent pricing pledge. |
| 5 | **Decision-maker (B2B / commercial / volume)** | Business hours. Volume needs. Wants contracts, invoicing, SLA, multi-location. | Replace with: property manager, IT procurement, multi-site operator, agency buyer. Copy: volume rates, multi-property / multi-seat / multi-location service, invoicing, SLA, dedicated rep. |

**Rule:** Each indexable page must address **at least 2 personas** explicitly in body or FAQ.

### How to pick personas per page

- **Emergency / urgent service page** ‚Üí Personas 1, 2, 4.
- **Cost / pricing page** ‚Üí Personas 4, 3, 5.
- **Buying guide / comparison** ‚Üí Personas 3, 4.
- **B2B / enterprise page** ‚Üí Personas 5, 4.
- **Local landing page** ‚Üí Personas 1 or 2 + 4.
- **How-to / troubleshooting** ‚Üí Personas 1 or 2 + 3.

## 5. GEO / AI Overviews ‚Äî 10-point checklist (per page)

Tick **all 10** before publishing:

- [ ] **Direct answer in first 100 words** ‚Äî declarative sentence answering the primary query verbatim. LLM extraction target.
- [ ] **Standalone paragraphs** ‚Äî each paragraph readable in isolation. No "as mentioned above" / "see below". LLMs extract paragraphs, not pages.
- [ ] **Question-phrased H2/H3** ‚Äî headings phrased as the questions users actually ask (match PAA). Not internal jargon.
- [ ] **TL;DR / Key Takeaways block** at top ‚Äî bulleted, scannable. 3‚Äì7 bullets max.
- [ ] **FAQ section** with FAQPage schema ‚Äî 3‚Äì6 questions matching `People Also Ask`.
- [ ] **Comparison tables** where applicable ‚Äî markdown-style tables get extracted well by LLMs.
- [ ] **Specific numbers** ‚Äî "1-year warranty", "from $X", "average Y-minute response" beat "fast" and "affordable". Concrete numerics are highly citable.
- [ ] **Citable expert quote** ‚Äî named expert with credentials, on the topic. Replace with `{expert_name}`, `{credentials}`.
- [ ] **`speakable` schema** targeting the direct-answer block + FAQ answers.
- [ ] **Last-updated date visible** + `dateModified` in schema.

### Why each point matters (one-liner each)

1. **Direct answer first** ‚Äî AI Overviews extract the first declarative sentence after the H1.
2. **Standalone paragraphs** ‚Äî LLMs index passages, not pages. A paragraph that needs context to make sense will not be cited.
3. **Question-phrased H2** ‚Äî AI matches user query to heading. Headings phrased as questions get matched more often.
4. **TL;DR** ‚Äî voice assistants and AI Overviews prefer scannable summaries.
5. **FAQ + schema** ‚Äî FAQPage schema directly powers AI Overview FAQ extractions.
6. **Comparison tables** ‚Äî LLMs parse tables structurally. A 5-row comparison table is highly citable.
7. **Specific numbers** ‚Äî numerics are quotable. "Fast" is not.
8. **Expert quote** ‚Äî adds attribution. Citable.
9. **Speakable** ‚Äî explicit signal for voice + assistant extraction.
10. **dateModified visible + schema** ‚Äî freshness signal. LLMs deprioritize content with no visible recency.

## 6. E-E-A-T ‚Äî 10-signal checklist (per page)

Per Google's December 2025 update, E-E-A-T applies to **all competitive queries**, not just YMYL. Every indexable page needs:

| # | Signal | Where to put it | Example |
|---|--------|-----------------|---------|
| 1 | **Author bio** with photo + credentials | Bottom of every page (use `AuthorBio` component) | `{expert_name}`, `{credentials}`, photo, bio paragraph |
| 2 | **Real first-party photos** | Used contextually in body, not just hero | Install photos, team photos, equipment photos ‚Äî not stock |
| 3 | **Customer quotes** with attribution | Trust section + scattered pull-quotes | First name + last initial + city/role |
| 4 | **Year founded** | Footer + about box on every page | "Since {year}" or "{N}+ years in business" |
| 5 | **License / cert / insurance** | Footer + contact page | License number, association membership, insurance carrier |
| 6 | **Warranty / guarantee** | Pricing, hero, FAQ | "X-year warranty on all work" |
| 7 | **Reviews count + score** | Hero badge + footer widget | "4.9 / 472 reviews" |
| 8 | **Local / niche proof** ‚Äî geo or sub-vertical specifics | Body paragraphs of landing pages | "Forest Hill garage repair" / "SaaS billing integration" ‚Äî concrete, not generic |
| 9 | **Editorial standards** statement | `/about/` page or privacy footer | How content is researched, who reviews, who approves |
| 10 | **Outbound citations to authorities** | Body, in-line | Industry association, standards body, gov source, peer-reviewed reference |

### Editorial standards ‚Äî what to include

The editorial standards page (or block in About) should answer:

- Who writes our content? (named authors with bios)
- How do we research? (data sources, primary research, expert interviews)
- Who reviews before publish? (named reviewer with credentials)
- How do we handle corrections? (correction policy + last-corrected date)
- What is our funding model? (transparent ‚Äî clients pay us; affiliate-free or affiliate-disclosed)

| **Job page** | `JobPosting`, `Organization` (hiringOrganization), `BreadcrumbList` | |

**Implementation:** centralize in `src/lib/schema.ts` or equivalent. Each page calls `localBusinessSchema(city)`, `articleSchema(post)`, etc. Don't hand-write JSON-LD per page ‚Äî that drifts.

**Validation:** every page ‚Üí Google Rich Results Test before publish.

## 8. Hub-spoke internal linking model

```
HOMEPAGE
  ‚Üì
SERVICE HUB A        ‚Üê‚Üí     SERVICE HUB B            (siblings cross-linked)
  ‚Üì                              ‚Üì
LOCAL/LANDING √ó N      LOCAL/LANDING √ó N             (one per city / region / segment)
  ‚Üì                              ‚Üì
DEEP CONTENT (blog / how-to / cost / comparison) clustered under each hub
```

### Rules

1. **Minimum 5 internal links per page** (enforced by lazy-method threshold).
2. **Anchor text variety** ‚Äî never repeat the same anchor 7 times on one source. Vary primary / secondary / natural-sentence anchors.
3. **Cross-link siblings** ‚Äî every local/landing page mentions 2‚Äì3 nearby siblings ("Also serving nearby {city_B} and {city_C}") with anchor links.
4. **Hub ‚Üí all spokes** ‚Äî the hub page links to every spoke. Spokes link back to the hub. Spokes link to 2‚Äì3 other spokes (not all ‚Äî keep it natural).
5. **Blog ‚Üí hub + spokes** ‚Äî blog posts link up to their parent hub + sideways to 2‚Äì3 relevant spokes.
6. **No weak anchors** ‚Äî "click here", "read more", "learn more" are flagged by lazy-method's internal-linking checker.
7. **No orphan pages** ‚Äî every indexable URL must be linked from ‚â•1 other indexable URL.
8. **Breadcrumb on every non-home page** ‚Äî `BreadcrumbList` schema + visible breadcrumb.

### Cluster depth ratios (rule of thumb)

- 1 hub : 10‚Äì20 spokes : 30‚Äì60 deep content pieces.
- If you have fewer spokes than 10, you don't have a hub ‚Äî you have a normal page.
- If you have more than 60 deep pieces per hub, split the hub.

### What we intentionally do NOT chase

- Arbitrary word counts unless tied to topic completeness.
- Keyword density tools' "perfect %" ‚Äî read-aloud test wins.
- Backlink count without context ‚Äî citations beat raw links in the AI era.
- AI-generated boilerplate paragraphs ‚Äî filtered by AI Overviews, hurts E-E-A-T.
- Excessive subdomain proliferation ‚Äî hurts topical authority.
- Schema for the sake of schema ‚Äî every schema block must reflect actual page content. Empty / fake schema is a Google manual-action risk.
